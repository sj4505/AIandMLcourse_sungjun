# SKILL: Vibe Coder Context Checkpoint System
# 외부 서비스 연결 오염 방지 + 롤백 프로토콜

---

## 이 스킬이 존재하는 이유

바이브 코더는 코드 자체보다 **외부 서비스 연결 구간**에서 세션이 무너진다.
에러가 나면 코드를 고치려 하지만, 진짜 원인은 대부분 **환경/설정 레이어**에 있다.

```
코드 레이어   → Claude가 잘 앎
환경 레이어   → Claude가 모름 (Vercel 대시보드, Polar 설정, OAuth 콘솔)
```

이 스킬은 세 가지를 강제한다:
1. 코드 수정 전 환경 레이어 먼저 검증
2. 검증 결과를 VERIFY.md에 기록 (텍스트 값으로)
3. 같은 것을 두 번 묻지 않음

---

## 트리거 조건

다음 중 하나라도 해당되면 이 스킬을 즉시 활성화한다:

```
- 401, 403, 404, 500 에러 등장
- "반영이 안 된다", "배포했는데 안 바뀐다"
- "왜 안되지", "이상하다", "분명히 했는데"
- Vercel / Polar / GitHub / Supabase / OAuth 관련 에러
- 같은 에러가 2회 이상 반복
- CLI 명령어 실행 전후
```

## 핵심 원칙

### 원칙 1. 코드 수정 금지 선언
에러 발생 시 코드를 수정하기 전에 반드시 환경 레이어를 먼저 확인한다.
환경 레이어 확인 없이 코드를 고치는 것은 원인을 모르는 채 처방하는 것과 같다.

### 원칙 2. "맞아/아니야" 금지 — 반드시 값을 직접 찾게 한다
Yes/No 대답은 받지 않는다. 반드시 **실제 값**을 사용자가 직접 찾아서 붙여넣게 한다.

왜 중요한가:
```
"portfolio 맞아요?" → "응"
= 확증 편향. 대충 봐도 맞는 것 같으면 "응" 하게 됨.
  실제로 안 봤을 가능성 높음.

"Production Branch가 뭐예요?" → 사용자가 직접 찾아야 함
= 찾는 과정에서 본인이 값을 직접 인식.
  틀렸으면 본인이 먼저 발견하게 됨.
```

### 원칙 3. 이유를 먼저 설명하고 확인을 요청한다
명령어나 확인 요청 전에 **왜 이걸 확인해야 하는지** 한 줄로 설명한다.
이유를 알면 사용자가 대충 넘어가지 않는다.
이유를 반복하면 사용자가 스스로 학습하게 된다.

```
❌ "vercel ls --prod 결과 붙여넣어줘"

✅ "지금 배포된 코드가 방금 푸시한 코드가 맞는지 확인해야 해.
    Vercel은 배포가 됐어도 실제로 다른 커밋이 올라가있을 수 있거든.
    VS Code 터미널에서 실행하고 결과 붙여넣어줘:
    vercel ls --prod"
```

### 원칙 4. 사용자가 직접 비교하게 만든다
값을 받은 후 Claude가 판단하되, 비교 과정을 사용자에게 보여준다.
"이 값이랑 저 값이 같으면 넘어가고 다르면 알려줘" 형태로 요청한다.
사용자가 직접 비교하면 다음엔 혼자 할 수 있게 된다.

### 원칙 5. 작은 확인마다 완료 선언
각 체크리스트 항목이 확인될 때마다 명시적으로 선언한다:
"✅ [항목] 확인됐어. 다음으로 넘어가자."
이렇게 하면 사용자가 진행 상황을 인식하고 대충 넘어가는 걸 방지한다.

### 원칙 6. 확인은 에러 발생 직전에
한 번 확인했다고 세션 내내 믿지 않는다.
CLI 명령어 실행은 환경을 덮어쓸 수 있다.

```
실제 사례:
  분명히 portfolio로 확인했는데 나중에 main으로 바뀌어있었음.
  원인: vercel CLI 명령어가 내부적으로 Production Branch를 main으로 초기화함.
  교훈: CLI 실행 전후로 반드시 재확인.

위험한 명령어 목록 (실행 전후 환경 재확인 필수):
  vercel          → 프로젝트 재연결 가능
  vercel link     → Production Branch main으로 초기화 위험
  vercel deploy   → 설정 덮어쓰기 가능
  vercel env pull → 로컬 env 덮어쓰기
```

### 원칙 7. VERIFY.md에 기록, 반복 질문 금지
한 번 확인된 값은 VERIFY.md에 기록한다.
이후 같은 항목을 다시 묻지 않는다. VERIFY.md를 근거로만 판단한다.
단, CLI 실행 후에는 예외적으로 재확인한다.

### 원칙 8. 모든 터미널 명령어는 VS Code 터미널 기준이다
Windows 환경에서 일반 cmd나 PowerShell은 Vercel CLI, git 등이
설치되지 않아 동작하지 않을 수 있다.

```
명령어 실행 위치:
  VS Code → 상단 메뉴 Terminal → New Terminal
  → 프로젝트 폴더가 열린 상태에서 실행

일반 cmd ❌
PowerShell (단독) ❌
VS Code 터미널 ✅
```

## 세션 시작 프로토콜

외부 서비스 연결 작업이 시작되면 즉시 실행한다.

### Step 1. 프로젝트 정보 고정
다음을 물어보고 VERIFY.md 최상단에 기록한다.
이 값들은 세션 내내 고정 기준점이 된다.
이후 모든 확인은 이 값 기준으로만 진행한다.

```
- Vercel 프로젝트명이 뭐야?
  (Vercel 대시보드 왼쪽 상단에 표시되는 이름)
- 배포 URL이 뭐야?
  (예: xxx.vercel.app)
- GitHub 레포명이 뭐야?
- 지금 작업하는 브랜치가 뭐야?
```

### Step 2. 현재 작동하는 것 체크포인트
작업 시작 전 현재 정상 동작하는 기능 목록을 기록한다.
이게 롤백 기준점이 된다.

```
VS Code 터미널에서 실행:
  git log --oneline -1
  → 현재 커밋 해시 기록 (롤백 시 이 커밋으로 돌아옴)

현재 작동 중인 것:
  [ ] 앱 로컬 실행 가능
  [ ] Vercel 배포 기본 동작
  [ ] (기타 이미 연결된 서비스)
```

## 서비스별 체크리스트

각 항목 확인 완료 시: "✅ [항목] 확인됐어. 다음으로 넘어가자." 선언 필수.

### Vercel 배포 에러 체크리스트

```
□ 1. 현재 브랜치 확인
      왜: 잘못된 브랜치에서 작업 중이면 아무리 고쳐도 반영 안 됨.
      VS Code 터미널: git branch --show-current
      값: ___

□ 2. Vercel Production Branch 확인
      왜: CLI 실행 시 이 값이 자동으로 바뀔 수 있음. 1번과 반드시 일치해야 함.
      위치: vercel.com → [프로젝트명] → Settings → Git
      값: ___
      → 1번과 일치해? 직접 비교해서 알려줘.

□ 3. 현재 Production 커밋 확인
      왜: 배포 성공해도 Ready Stale이면 현재 Production이 아님.
      VS Code 터미널:
        vercel ls --prod     ← Production 커밋 해시
        git log --oneline -1 ← 내 최신 커밋 해시
      → 앞 7자리 직접 비교해서 같으면 ✅, 다르면 알려줘.

□ 4. 환경변수 존재 확인
      왜: 변수 하나만 빠져도 인증/결제 전체 동작 안 함.
      VS Code 터미널: vercel env ls
      필요한 변수 전부 목록에 있어? 직접 확인해서 알려줘.

□ 5. 위 4개 모두 ✅ 후에만 코드 수정 진입
```

**⚠️ vercel 계열 명령어 실행 후 2번 반드시 재확인**
이유: vercel, vercel link, vercel deploy 실행 시
Production Branch가 main으로 자동 초기화될 수 있음.

### Google OAuth 에러 체크리스트

```
□ 1. 리다이렉트 URI 확인

      현재 Vercel 배포 URL: ___
      → https://[배포URL]/api/auth/callback/google 이 URI 목록에 있어?
        직접 비교해서 알려줘.

□ 3. 환경변수 4개 존재 확인
      왜: 하나라도 없으면 OAuth 전체가 동작 안 함.
      VS Code 터미널: vercel env ls
      - GOOGLE_CLIENT_ID: 있음/없음
      - GOOGLE_CLIENT_SECRET: 있음/없음
      - NEXTAUTH_SECRET: 있음/없음
      - NEXTAUTH_URL: 있음/없음

□ 4. 위 3개 모두 ✅ 후에만 코드 수정 진입
```

### Polar 결제/웹훅 에러 체크리스트

```
□ 1. 공식 SDK 사용 확인

          반드시 공식 SDK 써야 함. (직접 구현해서 삽질한 실제 사례 있음)

      @polar-sh/sdk 있어? 없으면 설치부터.

□ 2. 웹훅 URL 일치 확인
      왜: Polar에 등록된 URL과 실제 배포 URL이 다르면 웹훅이 아예 도착 안 함.
      위치: polar.sh → [프로젝트] → Webhooks
      등록된 URL 현재 값: ___
      현재 Vercel 배포 URL: ___
      → 일치해? 직접 비교해서 알려줘.

□ 3. POLAR_WEBHOOK_SECRET prefix 확인
      왜: Polar 시크릿은 반드시 polar_whs_ 로 시작해야 함.
          다른 prefix면 서명 검증 무조건 실패. (실제 삽질 사례 있음)
      Vercel 환경변수에서 POLAR_WEBHOOK_SECRET 값 앞 10자리: ___
      → polar_whs_ 로 시작해? 직접 확인해서 알려줘.

□ 4. 위 3개 모두 ✅ 후에만 코드 수정 진입
```

## 에러 루프 감지 + 롤백 프로토콜

### 에러 루프 감지 기준

```
같은 에러코드 2회 이상 등장
→ 즉시 선언:
  "지금 같은 에러가 반복되고 있어.
   코드를 계속 수정하면 컨텍스트만 오염돼.
   잠깐 멈추고 환경 레이어부터 다시 확인하자."
```

### 3회 연속 실패 시 롤백 프로토콜

```
Step 1. 시도 목록 정리
        왜: 다음 세션에서 같은 시도를 반복하지 않기 위해.
        "시도 1: ___ → 실패 (이유: ___)"
        "시도 2: ___ → 실패 (이유: ___)"
        "시도 3: ___ → 실패 (이유: ___)"

Step 2. 롤백 제안
        "3번 연속 실패했어.
         지금 컨텍스트에 실패한 시도들이 쌓여서
         계속 잘못된 방향으로 가고 있을 가능성이 높아.
         작업 시작 전으로 돌아가는 게 지금보다 빠를 거야.
         /clear 하고 다시 시작할까?"

Step 3. /clear 전에 VERIFY.md에 반드시 저장
        - 마지막 정상 커밋 해시: ___
        - 작동했던 것 목록: ___
        - 실패한 시도 목록 + 이유: ___
        → 이 파일이 다음 세션의 시작점이 된다.

Step 4. 새 세션 시작 시
        "VERIFY.md 읽어줘" 로 시작
        실패한 시도는 절대 반복하지 않음
        환경 레이어 체크리스트부터 다시 실행
```

## 절대 하지 않는 것

```
❌ 환경 레이어 확인 없이 코드 수정 시작
❌ "맞아요?" "확인했어요?" 형태의 Yes/No 질문
❌ 이유 설명 없이 확인 요청
❌ 같은 항목 두 번 이상 확인 요청 (CLI 실행 후 제외)
❌ Polar webhook 서명 검증 직접 구현
❌ CLI 실행 후 Production Branch 재확인 없이 진행
❌ 에러 루프 3회 이후 코드 계속 수정
❌ VS Code 터미널 외 다른 터미널 환경 가정
```

## Superpowers와 함께 사용 시

```
Superpowers 디버깅 스킬 실행 전에
이 스킬의 환경 체크리스트를 먼저 실행한다.
코드 레이어 진입은 환경 레이어 확인 후에만 허용.

역할 분리:
  Superpowers  → 코드를 어떻게 짤 것인가
  이 스킬      → 환경이 올바른 상태인가
```

## 이 스킬의 목표

```
Before:
  401 에러 → 코드 수정 → 실패 → 코드 수정 → 실패
  → 컨텍스트 오염 누적 → 세션 붕괴 → /clear → 처음부터

After:
  401 에러 → "왜 확인해야 하는지" 설명
           → 환경 레이어 체크리스트 (값 직접 찾기)
           → 원인 특정 → 정확한 수정 1회 → 해결
```

바이브 코더는 뭘 확인해야 하는지 모른다.
이 스킬은 확인해야 할 것을 **이유와 함께** 명령어와 URL로 정확히 알려주고,
사용자가 값을 직접 찾아서 비교하게 만들어,
Claude가 그 값으로 직접 판단한다.

반복할수록 사용자가 스스로 할 수 있게 된다.
이게 AI 의존도를 낮추는 방향이다.
